1. ¿Qué es k6 y para qué sirve?

k6 es una herramienta de pruebas de carga y rendimiento de código abierto desarrollada por Grafana Labs. A diferencia de herramientas como JMeter, k6 usa JavaScript para escribir los scripts y está optimizado para entornos de desarrollo modernos.

Los tests de k6 simulan múltiples usuarios reales (VUs — Virtual Users) accediendo a una API simultáneamente, permitiendo medir:

  • Cuánto tiempo tardan las respuestas bajo carga
  • Cuántos errores se producen
  • Cuál es el punto de saturación del sistema
  • Si el comportamiento funcional es correcto bajo presión
¿Qué es un VU (Virtual User)?
Un VU es un hilo de ejecución independiente que ejecuta el código del test en bucle. Con 100 VUs, es como si 100 personas estuviesen usando la aplicación al mismo tiempo, cada una haciendo peticiones sin esperar a las demás.

2. Objetivo del test

Este test está diseñado para ser difícil de pasar. Sus objetivos son:

Objetivo Qué se mide
Rendimiento bajo carga real Tiempos de respuesta con 80-120 usuarios simultáneos haciendo CRUD completo
Resistencia a un spike La API aguanta 250 usuarios de golpe solo leyendo durante 50 segundos
Escrituras intensas con ficheros 40 usuarios haciendo create+upload+update+delete sin pausa durante 90 segundos
Corrección funcional bajo presión Todos los endpoints siguen respondiendo con los datos correctos bajo carga extrema
Validaciones del backend El servidor sigue rechazando datos inválidos con 400 incluso bajo estrés

3. Arquitectura del test — visión general

El test está estructurado en tres capas:

Capa 1 — Configuración (options)

Define los escenarios, cuántos VUs tiene cada uno, cuánto duran y qué función ejecutan.

Capa 2 — Operaciones atómicas

Funciones individuales que hacen una sola cosa: listar(), crearCliente(), actualizarCliente(), etc. Cada una manda una petición HTTP, mide el tiempo y verifica que la respuesta sea correcta.

Capa 3 — Escenarios exportados

Las funciones que k6 ejecuta en cada VU: combinan las operaciones atómicas con pausas y probabilidades para simular comportamientos realistas.

4. Línea de tiempo: cómo transcurre el test

00:00 – 00:30 — Calentamiento
El escenario carga_mixta arranca y sube progresivamente de 0 a 30 VUs. El sistema se calienta, se establecen conexiones con PostgreSQL.
00:30 – 01:30 — Carga media
Los VUs suben de 30 a 80 y se mantienen durante 60 segundos. Tráfico mixto real: listados, creaciones, ediciones, cambios de estado y borrados.
01:30 – 02:00 — Carga alta
Los VUs de carga mixta escalan hasta 120. La API está bajo presión sostenida.
02:00 – 02:50 — ⚡ SPIKE: 250 VUs de golpe
El escenario spike_lectura arranca. En solo 10 segundos aparecen 250 VUs adicionales solo haciendo peticiones de listado. El sistema pasa de ~120 VUs a más de 350 VUs simultáneos. Este es el momento más crítico del test.
02:50 – 03:00 — Bajada del spike
Los 250 VUs de lectura se van apagando en 10 segundos. El sistema recupera algo de respiro.
03:00 – 03:15 — Carga sostenida alta
Solo queda el escenario mixto con 120 VUs. Continúa el CRUD completo.
03:15 – 04:45 — 🔥 Escrituras intensas + carga mixta
El escenario escrituras_duras arranca con 40 VUs constantes haciendo ciclos de create-con-logo → update → cambio de estado → delete sin apenas pausa. Esto estresa simultáneamente el sistema de ficheros (guardar/borrar PNGs) y PostgreSQL (INSERT/UPDATE/DELETE rápidos).
04:30 – 04:45 — Ramp-down
El escenario mixto apaga sus VUs gradualmente. Las escrituras duras terminan a los 4:45.
04:45 — Fin del test
k6 muestra el resumen final con todas las métricas y si los umbrales se han cumplido.

5. Los tres escenarios en detalle

Escenario 1 — carga_mixta

ramping-vus   120 VUs máx.   4m 30s

Simula el uso real de la aplicación: un usuario que entra, lista clientes, crea uno, lo edita y lo borra. Las etapas de la rampa son:

EtapaDuraciónVUs objetivoPropósito
Calentamiento30s30Conexiones iniciales, cachés de BD
Carga media60s80Tráfico normal de producción
Sostenida60s80Estabilidad bajo carga media
Presión alta30s120Subida a carga pico
Sostenida alta60s120Aguante bajo carga máxima
Ramp-down30s0Apagado ordenado

Cada VU ejecuta cicloMixto() que incluye:

  • Siempre: listar clientes con parámetros variados
  • 15% de probabilidad: probar validaciones (datos inválidos → 400)
  • 10% de probabilidad: buscar un ID inexistente (→ 404)
  • 30% de probabilidad: crear con logo (imagen real adjunta)
  • 70% de probabilidad: crear sin logo
  • Siempre: leer, actualizar y cambiar estado del cliente recién creado
  • 70% de probabilidad: borrar el cliente; 30% lo deja para que haya datos reales en los listados

Escenario 2 — spike_lectura

spike   250 VUs máx.   50s (desde min 2)

El escenario más agresivo. Simula un pico de tráfico repentino: 250 usuarios accediendo a los listados al mismo tiempo, como si se publicase un enlace viral o se lanzase una campaña de marketing.

Cada VU solo ejecuta soloLectura(): llama a listar() con una variante aleatoria de filtros, espera 50ms y repite. Las variantes de URL usadas son:

GET /api/clients?page=1&pageSize=10
GET /api/clients?page=2&pageSize=5
GET /api/clients?page=1&pageSize=25
GET /api/clients?page=1&pageSize=10&status=Active
GET /api/clients?page=1&pageSize=10&status=Prospect
GET /api/clients?page=1&pageSize=10&status=Inactive
GET /api/clients?page=1&pageSize=10&name=Corp
GET /api/clients?page=1&pageSize=10&name=Tech
GET /api/clients?page=1&pageSize=10&name=Startup&status=Prospect
GET /api/clients?page=3&pageSize=10
¿Por qué este escenario es tan duro?
Cada petición de listado hace dos queries a PostgreSQL: un COUNT(*) para calcular el total de páginas, y un SELECT … ORDER BY … SKIP … TAKE para traer los datos. Con 250 VUs simultáneos sin apenas pausa, PostgreSQL recibe ~500 queries concurrentes. Si la tabla tiene muchos registros acumulados de tests anteriores, las queries tardan más y el problema se amplifica.

Escenario 3 — escrituras_duras

constant-vus   40 VUs fijos   90s (desde min 3:15)

40 VUs ejecutando el ciclo más pesado posible sin pausa: crear un cliente con logo real → actualizarlo con otro logo → cambiar su estado → borrarlo. Todo seguido, sin descanso.

Esto estresa tres sistemas a la vez:

  • Sistema de ficheros: cada ciclo escribe un PNG en wwwroot/uploads/ y lo borra al final
  • PostgreSQL: INSERT → UPDATE → UPDATE → DELETE en bucle rápido
  • Parser multipart del servidor: dos peticiones con fichero adjunto por ciclo

6. Las operaciones atómicas — qué hace cada función

listar()

Hace un GET /api/clients con parámetros de paginación y filtros aleatorios. Verifica que la respuesta tenga status 200, sea JSON válido, contenga el campo data (array), totalItems, totalPages y que el número de elementos no supere el pageSize. Registra el tiempo en la métrica dur_listado_ms.

crearCliente()

Hace un POST /api/clients como multipart/form-data con nombre, descripción y estado aleatorios, sin logo. Para forzar el content-type correcto (el endpoint solo acepta multipart, no JSON), incluye un archivo vacío de 0 bytes que el servidor descarta porque comprueba logo.Length > 0. Verifica status 201, que el cuerpo tenga id numérico, name y logoUrl, y que la cabecera Location esté presente. Registra en dur_creacion_ms.

crearClienteConLogo()

Igual que el anterior pero incluye una imagen PNG real (1×1 píxel, ~68 bytes codificados en base64). Verifica adicionalmente que logoUrl no sea null y contenga /uploads/ en la URL. Registra en dur_upload_ms.

obtenerPorId(id)

Hace un GET /api/clients/{id}. Verifica status 200, que el id en la respuesta coincida con el solicitado, y que name y statusName sean strings.

obtenerIdInexistente()

Genera un ID aleatorio entre 9.000.000 y 9.999.999 (prácticamente imposible que exista) y hace un GET. Verifica que la respuesta sea exactamente 404 y que el cuerpo tenga el campo error. Importante: este 404 no se cuenta como error en tasa_errores porque es un comportamiento esperado y correcto.

actualizarCliente(id, estado)

Hace un PUT /api/clients/{id} como multipart con nuevos datos y un logo nuevo. Verifica status 200, que el id no haya cambiado, que updatedAt esté presente y que logoUrl no sea null (el logo se sustituyó). El logo anterior se borra del disco, el nuevo se guarda.

cambiarEstado(id, estadoActual)

Hace un PATCH /api/clients/{id}/status con JSON ({ "status": "Inactive" } por ejemplo). Elige siempre un estado diferente al actual. Verifica status 200 y que tanto status como statusName en la respuesta sean el nuevo estado. La operación más ligera del test, con el umbral más estricto (p95 < 400ms).

eliminarCliente(id)

Hace un DELETE /api/clients/{id}. Verifica status 204 y que el cuerpo sea vacío. Incrementa el contador total_borrados.

probarValidaciones()

Envía datos inválidos al servidor para verificar que las validaciones siguen funcionando bajo estrés:

  • Nombre de 1 carácter (mínimo son 2) → debe devolver 400
  • Descripción de 5 caracteres (mínimo son 10) → debe devolver 400

Ambas peticiones se envían como multipart (obligatorio) con un archivo vacío. No se cuentan en tasa_errores porque el 400 es la respuesta correcta esperada.

7. Métricas personalizadas

Además de las métricas que k6 recoge automáticamente, el test define las suyas propias para medir cada operación por separado:

Métrica Tipo Qué mide
dur_listado_ms Trend Tiempo de respuesta de GET /api/clients (con filtros y paginación)
dur_creacion_ms Trend Tiempo de respuesta de POST /api/clients sin logo
dur_upload_ms Trend Tiempo de respuesta de POST/PUT con logo real adjunto
dur_update_ms Trend Tiempo de respuesta de PUT /api/clients/{id}
dur_status_ms Trend Tiempo de respuesta de PATCH /api/clients/{id}/status
dur_delete_ms Trend Tiempo de respuesta de DELETE /api/clients/{id}
tasa_errores Rate Porcentaje de peticiones que fallan inesperadamente (excluye 404 intencionados)
total_creados Counter Número total de clientes creados con éxito durante todo el test
total_borrados Counter Número total de clientes eliminados con éxito durante todo el test

Los tipos Trend calculan automáticamente avg, min, max, p(90), p(95) y p(99). El tipo Rate calcula porcentajes. El tipo Counter acumula un total.

¿Qué es el p(95)?
El percentil 95 significa que el 95% de las peticiones fueron más rápidas que ese valor. Un p(95) de 400ms indica que solo el 5% de las peticiones tardaron más de 400ms. Es más representativo que el promedio porque no se ve afectado por valores extremos puntuales.

8. Umbrales (Thresholds) — cuándo el test pasa o falla

Los umbrales son los criterios de aceptación. Si alguno se incumple, el test termina con error (código de salida no cero) y la línea aparece en rojo con ✗.

Umbral Condición Qué significa si falla
http_req_duration p(95) < 800ms El 95% de TODAS las peticiones responden en menos de 800ms La API es demasiado lenta bajo la carga combinada de los tres escenarios
http_req_duration p(99) < 1200ms El 99% de las peticiones responden en menos de 1.2 segundos Hay demasiadas peticiones muy lentas, incluso en el 1% más lento
dur_listado_ms p(95) < 1200ms El 95% de los listados responden en menos de 1.2 segundos El endpoint de listado se satura bajo el spike de 250 VUs (COUNT + ORDER BY concurrentes)
dur_creacion_ms p(95) < 800ms El 95% de las creaciones responden en menos de 800ms La inserción en BD está lenta o el parsing multipart tarda demasiado
dur_upload_ms p(95) < 1500ms El 95% de las subidas con logo responden en menos de 1.5 segundos El sistema de ficheros está saturado o el servidor tarda en procesar el multipart
dur_update_ms p(95) < 600ms El 95% de los updates en menos de 600ms Las actualizaciones con logo son lentas bajo carga
dur_status_ms p(95) < 400ms El 95% de los cambios de estado en menos de 400ms Incluso la operación más simple (PATCH de un campo) es lenta — problema grave de concurrencia
dur_delete_ms p(95) < 400ms El 95% de los borrados en menos de 400ms Los deletes con borrado de fichero físico son lentos
tasa_errores < 1% Menos del 1% de peticiones fallan con error inesperado La API devuelve 415, 500 u otros errores que no deberían ocurrir
checks > 97% Al menos el 97% de todas las verificaciones pasan Las respuestas no tienen la estructura correcta o los datos son incorrectos
total_creados > 50 Se crearon más de 50 clientes durante todo el test El test no generó suficiente carga real de escritura

9. Setup y Teardown

setup()

Se ejecuta una única vez antes de que arranquen los escenarios. Hace una petición de prueba a GET /api/clients?page=1&pageSize=1 para verificar que la API está disponible. Si no responde con 200, muestra un aviso en consola. También imprime el banner de inicio con la URL y la duración estimada.

teardown()

Se ejecuta una única vez al finalizar todos los escenarios. Solo imprime un mensaje de finalización indicando que hay que revisar los umbrales en el resumen de k6.

10. Cómo ejecutar el test

Requisitos previos

  • k6 instalado (winget install k6 o descarga desde k6.io)
  • La API corriendo en http://localhost:5000 (dotnet run)
  • PostgreSQL corriendo con la base de datos clientmanager
  • La tabla Clients con pocos registros para que los listados sean rápidos

Comando de ejecución

# Desde la carpeta del test:
cd AplicacionAPI/ClientManager/ClientManager.API/k6-tests
k6 run stress-test-completo.js

Limpiar la base de datos antes de ejecutar

Es importante ejecutar esto en PostgreSQL antes de cada test para evitar la acumulación de registros de ejecuciones anteriores (que degradan el rendimiento del listado):

-- Borra todos los clientes generados por tests, conserva los originales (IDs 1, 2 y 3)
DELETE FROM "Clients" WHERE "Id" NOT IN (1, 2, 3);

Qué mirar primero en los resultados

Lo que ves Lo que significa
✓ checks rate=100% La API se comporta funcionalmente correcta bajo toda la carga
✓ tasa_errores rate=0.00% Cero errores inesperados — la API aguanta sin lanzar 500 ni 415
✗ dur_listado_ms p(95)>1200ms El listado se satura bajo el spike. Causas: tabla con muchos registros acumulados, connection pool lleno, o máquina sin recursos
http_req_failed: 4.4% Normal y esperado: son los 404 de obtenerIdInexistente() que k6 cuenta como "fallidos" pero que el test no considera errores

http_reqs: ~350/s Throughput total del sistema: ~350 peticiones HTTP procesadas por segundo
Resultado ideal
Todos los thresholds en verde (✓), checks rate=100%, tasa_errores rate=0.00%, y total_creados > 15000. Eso indica que la API maneja correctamente más de 350 peticiones/segundo durante 5 minutos continuos con carga mixta, spike de lectura y escrituras intensas simultáneamente.
