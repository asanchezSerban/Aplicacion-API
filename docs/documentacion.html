Arquitectura

Modelo de negocio y estructura del sistema

ClientManager es una aplicación multitenant de gestión interna. El SuperAdmin gestiona empresas y clientes; cada Cliente solo puede ver sus propios datos.

Modelo de negocio

SuperAdmin
├── Gestiona Companies (empresas)
│   ├── Acme Corp           [Active]
│   └── Tech Startup SL     [Prospect]
│
└── Gestiona Clients (personas asignadas a empresas)
    ├── Cliente A  →  pertenece a Acme Corp
    └── Cliente X  →  pertenece a Tech Startup SL

Cada Client puede autenticarse y ver únicamente sus propios datos.

FASE 02

Identity + JWT

Sistema de autenticación con tokens firmados

Se integra ASP.NET Core Identity para la gestión de usuarios y contraseñas, y JWT Bearer para la autenticación stateless mediante tokens firmados.

Flujo de autenticación

1
POST /api/auth/login El usuario envía email y contraseña
2
Verificación de lockout Si el usuario tiene ≥5 intentos fallidos, la cuenta está bloqueada 15 minutos
3
Validación de contraseña PBKDF2-HMAC-SHA512 · Si falla, incrementa el contador de intentos fallidos
4
Generación del JWT HMAC-SHA256 · Expiración: 15 minutos · Claims: sub, email, jti, role, clientId
5
Respuesta Devuelve accessToken, expiresAt, userEmail, role

Política de contraseñas

RequisitoValor
Longitud mínima8 caracteres
Requiere mayúsculasSí
Requiere minúsculasSí
Requiere dígitosSí
Requiere carácter especialSí
Intentos antes de lockout5
Duración del lockout15 minutos

FASE 03

Autorización por roles

Control de acceso basado en el rol del usuario autenticado

Todos los endpoints quedan protegidos. Un token válido no es suficiente — el rol que contiene debe coincidir con el requerido por el endpoint.

Roles y permisos

RolAcceso permitido
SuperAdminCRUD completo de /api/companies y /api/clients
ClienteSolo GET /api/clients/me — sus propios datos
Sin token401 Unauthorized en todos los endpoints protegidos
Token con rol incorrecto403 Forbidden

Endpoint exclusivo del Cliente

GET /api/clients/me Datos propios del cliente autenticado — lee el claim clientId del JWT Cliente

Casuística de respuestas

SituaciónRespuesta
Sin token en la petición401 Unauthorized
Token expirado401 Unauthorized
Token válido pero rol incorrecto403 Forbidden
Token válido y rol correcto200 OK con los datos
FASE 04

Refresh tokens

Renovación de sesión sin re-login

Se implementa un sistema de dos tokens para equilibrar seguridad y comodidad: el access token (JWT de corta vida) y el refresh token (opaco, revocable, de 24 horas).

Comparativa de tokens

Access Token (JWT)

Duración15 minutos
TipoJWT firmado
AlmacenamientolocalStorage
ContieneuserId, email, rol
ValidaciónFirma criptográfica
RevocableNo (hasta que expira)

Refresh Token

Duración24 horas
TipoUUID aleatorio
AlmacenamientoCookie HttpOnly
ContieneNada (opaco)
ValidaciónConsulta en BD
RevocableSí, inmediatamente

Flujo de renovación

1
Access token expira (15 min de inactividad) La API responde 401 a la siguiente petición
2
Interceptor Angular detecta el 401 Llama automáticamente a POST /api/auth/refresh (Fase 5)
3
API valida el refresh token en BD Comprueba que existe, no está revocado y no ha expirado
4
Rotación de tokens El refresh token antiguo se revoca · Se generan nuevos access + refresh tokens
5
Interceptor reintenta la petición original El usuario no percibe la interrupción

Tabla en BD

CampoTipoDescripción
TokenVARCHAR(200) UNIQUEUUID aleatorio — índice único para búsqueda rápida
UserIdVARCHAR FK → AspNetUsersPropietario del token
ExpiresAtTIMESTAMPTZExpiración: creación + 24h
RevokedAtTIMESTAMPTZ NULLMomento de revocación — null si está activo
ReplacedByTokenVARCHAR NULLToken que lo reemplazó al rotar — útil para auditoría

Nuevos endpoints

POST /api/auth/login Devuelve access token + refresh token (cookie HttpOnly) público
POST /api/auth/refresh Rota el refresh token y devuelve nuevos tokens público
POST /api/auth/logout Revoca el refresh token activo · Cierra sesión autenticado
SEC

Capas de seguridad implementadas

Defensa en profundidad — cada capa protege contra una amenaza distinta

Política de contraseñas

Mínimo 8 chars, mayúsculas, minúsculas, dígito y carácter especial obligatorios

Hashing PBKDF2

PBKDF2-HMAC-SHA512 con sal única por usuario — Identity gestiona automáticamente

Bloqueo por intentos

5 intentos fallidos → bloqueo de 15 minutos — protege contra fuerza bruta

Anti-enumeración de usuarios

Mismo mensaje de error para usuario no encontrado y contraseña incorrecta

JWT de corta vida

Access token válido solo 15 minutos — reduce la ventana de uso si es robado

Refresh token revocable

UUID opaco almacenado en BD — se puede revocar inmediatamente en logout

Cookie HttpOnly

Refresh token en cookie HttpOnly + Secure — JavaScript no puede accederlo

Autorización por roles

SuperAdmin y Cliente con permisos distintos — token válido no garantiza acceso

Secrets fuera del repositorio

JWT SecretKey y contraseñas en User Secrets — nunca en appsettings.json
